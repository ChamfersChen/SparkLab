## 关键技术决策

1. **作者可见性 404 而非 403** — 不泄漏资源存在性；管理员可走 `/api/admin/templates/{id}` 看到任意草稿。
2. **变量覆盖校验只在 create / 双方都传 update 时触发** — `update_template` 当 `variable_hints` 和 `input` 同时显式传才校验；只改一边时不强求（plan 决策保持保守）。
3. **Pydantic 422 在前端解析** — `formatErrorMessage` 检测 `Array.isArray(detail)`，按 `field: msg` 拼成可读 message；`detail` 为对象时回退 `message/error`；字符串直接展示。
4. **`apiAdmin*` 自动加 /admin 前缀** — 与后端真实的 `/api/admin/*` 路径对齐；调用方传 `'/templates'` 即可；保留 `apiSuperAdmin*` 现状（多个子资源路径不规则）。
5. **`fill` / `use` 接口也走可见性** — 草稿模板不能填、不能计数使用；与 `GET /:id` 行为一致。
6. **`get_template` 不再修改 instance.variable_hints** — 之前是 setattr dict 解析 JSON，污染 SQLAlchemy session 的 dirty 状态，导致 PUT 阶段触发 autoflush 把 dict 当 str 写库 DataError。改由 Pydantic 响应模型 `parse_hints` 字段 validator 解析。
7. **TemplateManage 的 DELETE 与 changeStatus(archived) 二选一** — 后端两者都软删成 archived，UI 上保留一份 changeStatus 即可，删除另一份避免重复按钮 + dead code。
8. **前端 watch debounce 500ms** — `a-input-search` 自身有 input 防抖，但填写表单是 `<a-textarea>`，每次按键都序列化整个 formValues 写 localStorage 浪费。直接 500ms debounce 解决。

## 复盘

- **pre-existing 错误 vs 新增错误** — 跑 lint 时严格区分，本次 0 个新增；改动边界清晰。
- **后端 PUT 阶段 DataError 修复路径** — 起点是 service 层 `get_template` 在 instance 上 setattr，污染 SQLAlchemy dirty 状态。这种"读路径修改 instance 属性"的副作用在 SQLAlchemy 异步 session 里很容易踩——记住：响应解析交给 Pydantic 字段 validator。
- **测试间状态不隔离** — conftest 没有 rollback fixture，导致 username 撞库。后端 e2e 想做"每个测试独立"，需要加 `db` session fixture + `BEGIN; ROLLBACK;` 包每个测试，或者重置唯一 key（username 加 uuid）。
- **`apiAdmin*` 改自动加 /admin 前缀** — 一开始没看出 base.js 这个设计是"反范式"（约定调用方拼前缀），改之后反而减少了 6 处易错点；这印证了"约定不如配置"的判断。
